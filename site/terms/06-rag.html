RAG (Retrieval-Augmented Generation)

An LLM is trained on data up to a certain date. It does not know what happened yesterday. It does not have access to private documents or proprietary knowledge. RAG fixes this by letting the model look things up before answering.

Here is how it works. Documents get broken into chunks. Each chunk is converted into a mathematical fingerprint called an embedding that captures meaning, not just words. These get stored in a vector database. When someone asks a question, the system finds the chunks with the closest meaning and places them into the context window. The model writes its answer with the reference material right in front of it.

The chunk-context problem: When you chop a document into chunks, each loses its surrounding context. "Revenue grew 3%" is useless alone. Anthropic's contextual retrieval prepends a brief explainer to each chunk before indexing. This reduced retrieval failures by 35%. Combined with keyword matching and reranking, failures dropped by 67%.

RAG vs. fine-tuning: Fine-tuning rewires the model itself -- expensive, slow, hard to update. RAG leaves the model untouched and feeds it reference material at query time. Documents are swappable. For most enterprise use cases, RAG is the right starting point.
